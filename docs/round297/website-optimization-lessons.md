# Round297 Website Optimization Lessons

Version target: `round297-real-qa-account-readiness-ledger-20260614`

Round297 turns repeated release friction into four concise rules that can survive the next rounds.

## Public Proof

- Treat worker-local checks as route material until the public shell, source app, browser evidence, and visible version marker agree.
- Name `lghui.top` when proving the public entry chain; do not substitute a local or source-only proof.
- Keep pending release integration explicit when the main thread has not rerun the full public gate.

## Auth Boundary

- `lghui.top` public-shell proof and `pages.dev` authenticated browser proof answer different questions.
- Real QA account claims require readiness plus browser gates; missing credentials stay `blocked/not-run`.
- Private-video management additionally requires `FM_PRIVATE_MEDIA` R2 before storage-backed production recovery can be claimed.

## JSON Budgets

- Keep public JSON growth under 250KB per round unless the ledger is lazy-loaded and gzipped.
- Require gzip byte-equivalence for generated ledgers.
- Put heavy audit detail behind intent-driven links so the homepage remains a learning surface, not a data dump.

## UI Visibility

- User-facing pages must show the counts and blockers they depend on: 316/61, 38/30/61, account readiness, and private-video blocked state.
- Hidden JSON-only fixes are not enough for release language.
- Blocked states should use the same words in UI, runbook, JSON, and final report.
